# Seele v0.5 框架锐评报告

> 基于 v0.5 双网关重构完成后全量分析

## ✅ 框架优点

### 1. 架构清晰度

**双向解耦做得好。** `agent/`（编排）→ `contexts/`（会话）→ `provider/`（外部实现）的树形依赖严格执行，从任何一个节点出发沿 import 方向走都不会产生循环。双 Gateway 的分离（API Gateway 管号池路由、Tool Gateway 管工具路由）让单个 Gateway 职责单一，不会被"上帝 Gateway"问题腐蚀。

### 2. 装配件模式（Plugin）

Plugin 放在 `tool/holder/` 而非 session 层是正确的——工具分组是注册中心的职责，与会话状态无关。Include/Exclude + glob 的模式提供了足够的灵活性，覆盖了绝大多数场景的白名单/黑名单需求。Clone 机制让 Agent 能向 Session 传播共享定义、同时保持独立激活状态。

### 3. 接口提取（tool/interfaces）

把 `ToolProvider`/`ToolHandler`/`ToolEntry` 抽到独立的 `interfaces/` 子包解决了"provider 包既定义接口又实现"的历史耦合问题。现在 Hub、MCP、内联三种 provider 各自只依赖 `interfaces/`，不存在 provider 包内部的隐式耦合。

### 4. 号池抽象（agent/api）

`AccountPool` + YAML 配置 + round-robin 提供了基础的负载均衡能力。`WithAccountPool()` 的链式调用设计让 LLM 客户端可以选择是否使用号池，不影响现有使用 `LLMConfig` 单账号的调用方。

### 5. Inline 直接注册

删除 `InlineProvider` 包装、改为 `Holder.RegisterInline()` 是正确的方向——内联工具本质上就是工具，不应该披一层 "Provider" 的外衣来增加心智负担。

## ⚠️ 缺点与隐患

### 1. 测试覆盖严重不足

有测试的包仅有 `test/`（集成）、`provider/`（已删除）、`provider/account/`（已删除）、`agent/tool/`（SchemaOf 测试）、`workplan/`。核心的 `agent/` `agent/api/` `agent/gateway/` `contexts/` `contexts/history/` 等包**零测试**。Gateway 接口无 mock 实现，无法上单元测试。

**风险**：后续改动 Gateway 路由逻辑时，只能依赖冒烟测试，无法快速定位回归。

### 2. Session/ToolDispatcher 接口仍是隐式耦合

`contexts/interface.go` 中的 `ToolDispatcher` 接口要求实现 `Tools()` 和 `Dispatch()`。`toolgw.Gateway` 接口也定义了同样的一组方法（加上 Plugin 控制）。两者之间**没有正式的接口契约**——`toolgw.DefaultGateway` 恰好满足了 `ToolDispatcher` 是因为方法签名碰巧匹配，而非通过显式的 `var _ ToolDispatcher = (*DefaultGateway)(nil)` 断言。

**建议**：`toolgw.Gateway` 应该内嵌或扩展 `seelectx.ToolDispatcher`。

### 3. Plugin 激活是全局的

当前 `Holder.ActivatePlugin()` 是全局状态——一个 session 激活 "dev"，所有 session 都受影响。虽然 Holder 的 pluginMgr 可以被 Clone，但 Agent 到 Session 的传播是值复制时机问题。如果两个 session 共享同一个 Holder，它们共享 Plugin 状态。

**建议**：Plugin 激活状态应该与 session 绑定，而非与 Holder 绑定。

### 4. 错误处理粒度不一致

工具调用的错误处理有三种方式：
- Hub 返回 `ErrToolUnavailable`（瞬时重试）
- MCP 返回 `ErrToolUnavailable`（连接断开）
- 内联工具返回自定义 error（无重试）

但 `dispatchToolCalls` 中所有 dispatch 失败都被 JSON 包裹后注入 history，LLM 无法区分"工具不存在"和"参数错误"。**所有错误都被吞掉了。**

### 5. go.mod 依赖膨胀

本来只需要 HTTP client 的框架，现在依赖了 `microHub`（gRPC）和 `mcp-go`（MCP client）。这两个依赖只在特定 provider 中使用，但在编译时会被拉入主模块。

**建议**：考虑用 build tags 或 plugin 机制让 MCP/Hub 成为可选依赖。

## 🔥 最大疏漏：缺少错误边界与超时传递

**整个框架没有统一的 `context` 超时传播策略。**

具体表现：

1. `agent.New()` 接受 `ToolCallTimeOut` 参数，但这个超时只传给 `HubToolHandler`。`MCPToolHandler` 和 `InlineToolHandler` **完全不使用这个超时**——它们使用 `context.Background()` 或依赖 HTTP 客户端的默认超时。

2. `dispatchToolCalls` 在 `contexts/dispatch.go` 中并发执行所有 tool_call，但**没有任何全局超时**控制。如果某个工具 hang 住（例如 MCP Server 无响应），整个 session 的 Chat 循环会卡死，直到达到 `MaxLoops` 上限。

3. `chatLoop` 中对 `completionStrategy.execute()` 的调用没有超时包装——如果 LLM 客户端 hang（网络问题、DNS 问题），Chat 方法会永久阻塞。

4. Race detector 在 `contexts/cache` 包中发现了竞态条件（`TestFileCache_*` 全组失败），说明并发安全的设计没有贯彻到底层组件。

**这是最致命的疏漏**——一个编排框架如果在编排层自身不管理超时和错误边界，上层应用就无法构建可靠的 ReAct 循环。所有工具调用的 context 都应该有一个**继承自 session 创建时传入的父 context 的派生超时**，而不是各自为政。

### 修复优先级

```
P0:  统一 dispatch 的超时传递 → 所有 handler 从 context 继承 deadline
P0:  Agent 级全局 dispatch 超时 → 覆盖 chatLoop 的 tool_call 阶段
P1:  显式 ToolDispatcher 接口断言 → toolgw.Gateway 内嵌 ToolDispatcher
P1:  Plugin 激活绑定 session
P2:  MCP/Hub 可选依赖（build tags）
P3:  新增包测试覆盖
```
